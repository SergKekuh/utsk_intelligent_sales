Часть 0.1 SQL 1 — Структура базы данных

🎯 Для чего это делается и что сделано

Проблема, которую решаем: До начала проекта вся информация о клиентах, товарах и продажах хранилась разрозненно — в бумажных отчётах, Excel-файлах, в учётной системе 1С. Менеджер тратил 10-15 минут, чтобы собрать данные по одному клиенту: найти его прошлые заказы, понять, что он обычно покупает, какой у него статус.

Что сделано: Создана единая база данных из 14 таблиц, куда загружены реальные сведения: 556 клиентов, более 3 тысяч товаров, почти 3 тысячи документов о продажах на сумму 210 млн грн. Все таблицы связаны между собой — система «понимает», какой клиент что купил, когда и на какую сумму.

💼 Ценность для бизнеса
  • Экономия времени менеджера: полная история клиента доступна за 2 секунды вместо 15 минут поиска
  • Автоматическое определение статуса клиента: система сама помечает, кто «активный», кто «спящий», а кто «ушедший»
  • Основа для будущей автоматизации: рекомендации, дашборды и отчёты работают на этих данных
🔧 Как это устроено (простыми словами)
  • Справочники — таблицы-словари: направления деятельности (Агро, Строй, ЖКХ...) и правила расчёта статусов клиентов
  • Основные сущности — товары (трубы, фитинги, отводы) и клиенты с их характеристиками
  • Транзакции — документы о продажах и строки в них (какой товар, сколько, на какую сумму)
  • ML-блок — таблицы для «умных» функций: кросс-продажи (что с чем покупают), самообучение рекомендаций, учёт отказов
  • Представления (Views) — готовые витрины данных: рекомендации для менеджера и дашборд с ключевыми показателями

Часть 0.2 SQL 2 — Расширение БД (рекомендации по размерам)

🎯 Для чего это делается и что сделано

Проблема, которую решаем: Компания продаёт несколько тысяч видов труб разных размеров. Менеджер физически не может помнить все размеры и их соответствия. Если клиент купил трубу 120×80×6 мм, менеджер вряд ли предложит ему похожую трубу 120×80×8 мм (толще стенка — выше цена — больше маржа).

Что сделано: Мы «научили» базу данных понимать размеры труб прямо из названий. Создана функция, которая автоматически извлекает диаметр, толщину стенки, размеры профиля. По этим размерам система находит похожие трубы и рекомендует их менеджеру.

💼 Ценность для бизнеса
  • Увеличение среднего чека: система предлагает усиленную версию трубы (толще стенка — дороже товар)
  • Менеджер не обязан быть «ходячим справочником»: ему не нужно помнить сотни размеров — система подсказывает
  • Клиент получает предложение «как для себя»: ему рекомендуют именно то, что подходит под его задачи
🔧 Как это устроено (простыми словами)
  1. Парсинг названий: программа «читает» название «Труба проф. 120×80×6» и понимает: ширина 120 мм, высота 80 мм, стенка 6 мм
  2. Расчёт веса: по формуле вычисляется вес одного метра трубы
  3. Поиск соседей: создана таблица связей, где для каждой трубы указаны её «ближайшие родственники» по размерам
  4. Генерация рекомендаций: когда менеджер открывает карточку клиента, система показывает: «Клиент брал 120×80×6 — предложите 120×80×8»

📐 Пример работы

Пример: Клиент купил Труба проф. 120×80×6 → система предлагает Труба проф. 120×80×8 (та же ширина, толще стенка) и Труба проф. 140×80×6 (похожий профиль).

Часть 1 Организация проекта и Git

🎯 Для чего это делается и что сделано

Проблема: Если код программы хранится только на одном компьютере — это риск. Поломка жёсткого диска, вирус, случайное удаление файла — и весь проект потерян. Если над проектом будут работать несколько человек — им нужно общее хранилище и понятная структура файлов.

Что сделано:
• Создана чёткая структура папок (как в хорошей библиотеке — у каждой книги своё место)
• Проект загружен на GitHub — защищённые сервера, доступные из любой точки мира
• Настроена система Git — «машина времени», которая запоминает каждое изменение. Если что-то сломалось — всегда можно вернуться к вчерашней версии

💼 Ценность для бизнеса
  • Сохранность кода: проект не пропадёт — хранится на серверах GitHub с тройным резервированием
  • История изменений: всегда видно, кто, когда и зачем внёс правки
  • Быстрое подключение: новый разработчик клонирует проект за 2 минуты

Часть 3 Управление зависимостями

🎯 Для чего это делается и что сделано

Часть 6 Сервисный слой

🎯 Для чего это делается и что сделано

Проблема: Менеджерам нужны готовые инструменты для работы, а не «сырые» запросы к базе: «найди клиента по ИНН», «покажи все продажи за март», «кто из клиентов перестал покупать».

Что сделано: Созданы три сервиса:
• ClientService — поиск клиентов, расчёт статусов, история покупок
• ProductService — каталог товаров, поиск по размерам, наличие на складе
• AnalyticsService — анализ продаж: топ-товаров, динамика по месяцам, выручка по клиентам

💼 Ценность для бизнеса
  • Поиск клиента за 1 секунду по названию, коду или ИНН
  • Мгновенный список «спящих» клиентов — кому пора позвонить
  • Анализ продаж: что продаётся лучше всего, какая динамика по месяцам

Часть 6.5 Сервис умных рекомендаций

🎯 Для чего это делается и что сделано

Проблема: Менеджер открывает карточку клиента и думает: «Что бы ему ещё предложить?» Он не помнит всю историю покупок клиента, не знает о новых поступлениях в категории клиента, не видит связей между товарами.

Что сделано: Создан «движок рекомендаций», который анализирует историю покупок клиента и выдаёт персонализированный список из 5 товаров. Система учитывает четыре блока и самообучается на реакциях менеджера.

💼 Ценность для бизнеса
  • Блок «История»: «Вы покупали это раньше — пора повторить заказ»
  • Блок «Новинки»: «Появилась новая позиция в вашей категории»
  • Блок «Cross-sells»: «С этой трубой обычно берут фланцы и отводы» (увеличение чека)
  • Блок «Похожие размеры»: «120×80×6, а есть усиленная 8 мм»
  • Самообучение: если менеджер принимает рекомендацию — её вес растёт (+5), если клиент отказался — падает (-3). При отрицательном весе товар больше не предлагается

Часть 7 UI — Консоль + GUI

🎯 Для чего это делается и что сделано

Проблема: Нужно два варианта интерфейса. Консольный — быстрый, для опытных пользователей, работает даже на слабых компьютерах и через удалённый доступ. Графический (GUI) — удобный, с кнопками и таблицами, понятный любому сотруднику.

Что сделано:

• Оба варианта используют одни и те же сервисы — функциональность полностью идентична

💼 Ценность для бизнеса
  • Консольный режим: мгновенный запуск, работает через SSH на сервере
  • Графический режим: интуитивно понятен — не требует обучения
  • Единая функциональность: что в консоли, что в GUI — результат одинаковый

Часть 8 Тесты, CI/CD, Docker, Документация

🎯 Для чего это делается и что сделано

Проблема: Когда программа разрастается, любое изменение может случайно сломать то, что уже работало. Развернуть программу на новом сервере раньше было целой историей: установить все зависимости, настроить базу данных, прописать пути.

Что сделано:

💼 Ценность для бизнеса
  • 46 автоматических тестов: перед каждым обновлением проверяется, что ВСЁ работает правильно. Исключены «сюрпризы» при запуске новой версии
  • CI/CD (GitHub Actions): при любом изменении кода тесты запускаются автоматически. Разработчик узнает об ошибке через минуту, а не через неделю от пользователей
  • Docker-контейнер: программа упакована со всеми настройками в «контейнер». Развернуть на новом сервере — 5 минут и одна команда. Не нужно вручную ничего настраивать
  • Документация: полное описание системы для разработчиков + руководство для пользователей (DIRECTOR_GUIDE)
🔧 Что происходит при изменении кода (автоматически)
  1. Разработчик вносит изменение и отправляет на GitHub
  2. GitHub Actions автоматически запускает 46 тестов
  3. Проверяется: работа с БД, рекомендации, веб-интерфейс, консоль
  4. Если все тесты прошли — код готов к использованию
  5. Если есть ошибка — разработчик получает уведомление и сразу исправляет
